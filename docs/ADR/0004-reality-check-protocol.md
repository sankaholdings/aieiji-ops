# ADR-0004: 実態確認プロトコル（Reality Check Protocol）

- **Status**: Accepted
- **Date**: 2026-04-27
- **Decider**: 三箇栄司

## Context（背景）

ADR-0002（2026-04-27）で「メモリと実態のズレ」を初めて記録した。同日、**全く同じ種類のミスが再発**した:

- メモリ「Phase 2はApify経由でTripAdvisor/Booking等に接続」を信じてApifyアカウント登録〜トークン取得まで進めた
- 実際は claude.ai にAnthropic公式の Booking.com / Tripadvisor / Resy / Viator コネクタが既に提供されており、Apifyは不要だった
- 社長の時間を無駄にし、ストレスを生じさせた

これは **「メモリ・計画を鵜呑みにして、現実の状態を確認せずに実行する」アンチパターン**である。1日に2回も発生したことは、個別ミスではなく**プロセス上の欠陥**を示している。

## Decision（決定）

すべての新規機能導入・外部サービス連携・Knowledge追加作業の前に、**実態確認プロトコル**の実行を必須とする。

### 必須チェックリスト（Reality Check）

着手前に**必ず**以下を確認し、結果をユーザーに報告する：

1. **claude.ai 設定 → コネクタ画面の現状確認**
   - 既に公式コネクタで代替できないか
   - 既に類似コネクタが接続済みでないか
2. **対象 Claude.ai Project の現状確認**
   - 手順（システムプロンプト）の中身
   - Knowledge にアップロード済みのファイル
   - 接続済みコネクタ
3. **ローカル資産の確認**
   - `05_Concierge/` 等、既存ファイル群に該当資料がないか
   - 既存スクリプト・MCPで代替できないか
4. **上記3つで「既存物で代替不可」が明確になってから、新規導入を提案する**

### 違反時の対応

このプロトコルを実行しないまま外部サービス登録・トークン発行・有料サービス契約を社長に促した場合、**重大ミス**として扱う:

- 即座に作業を停止し謝罪
- ミス内容と再発防止策をADRに追記
- ユーザーの判断を仰ぐ

## Consequences（結果）

### 良い結果
- 「既に揃っているもの」を見落とさない
- 社長の時間・精神的負荷の浪費を防ぐ
- メモリと現実のズレを早期発見できる

### トレードオフ
- 着手前の確認ステップが増える（数分のオーバーヘッド）
- ただし、本日のApify騒動は**1時間以上**を消費したので、Reality Checkの数分は明らかに割に合う

### 適用範囲
- Claude.ai Projects 関連作業（Knowledge・コネクタ・手順）
- 外部サービス登録・有料サービス契約
- 1106PC orchestrator 関連の新規Issue起票
- ローカルスクリプト新規作成
