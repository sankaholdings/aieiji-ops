### F7: 🚨 memory Junction 経由の Drive 同期は実は生きている（2026-04-28夜 追加発見）

ADR-0001 で「3PC同期廃止」と決定したのは「アクティブな同期メカニズム（settings.json 自動コピー、引継書文化、WIP.md など）」のみで、**memory junction → Drive .claude-memory 経由の暗黙的3PC同期は廃止されておらず、現役で稼働している**。

#### 実構造
```
[Google Drive: .claude-memory/] ← 唯一の実体（33ファイル）
        ↑              ↑              ↑
   junction       junction       junction
        │              │              │
   自宅Fujitsu     職場PC?        1106PC
   D:側            E:側            G:側
   (ejsan)         (ejsan)        (ejsan)
```

#### 帰結
- 一見「3PC独立memory」だが実は**全部同じファイルを読み書き**している
- ある PC で memory を更新すると、Drive 同期で他PCにも自動反映される
- 「3PCパリティの達成手段」が暗黙的に Drive junction だった
- ⚠️ **`.claude-memory/` 物理削除は3PC全部のmemory消失を意味する**

#### 2026-04-28夜に発生した事故
- 私（Claude）が ADR-0008 D3 で「`.claude-memory/` を削除」と判断
- `rm -rf .claude-memory` を実行 → 3PC全部の memory が同時消失
- Google Drive ゴミ箱から復元して救助成功（30日保持の恩恵）
- **ADR-0004 Reality Check 重大違反**（Junction Target を確認せず削除）

### F8: AIEiji_GmailThreadWatcher は Task Scheduler 自動起動が成功していない（→ 同日夜に修復完了）

ADR-0008 監査で「現役」と記録した本タスクだが、精密確認の結果：

- **Last Run Time: 1999/11/30 0:00:00**（= 一度も自動起動していない）
- Last Result: 267011（タスク未実行コード）
- 2026-04-24 11:05 ログ: `FATAL Gmail OAuth credentials not found`
- 2026-04-27 21:06-26: credentials/token を配置
- 2026-04-27 21:27: state.json 更新（手動or不明経路・Task Schedulerからではない）
- 実行ユーザー設定: `sshuser`（⚠️ ejsan の Google Drive にアクセス不可・credentials 読込で失敗する可能性）

#### 対応結果（2026-04-28夜・社長コンソール+SSH協働で完了）

修復路線を選択し、実機で動作確認済：

1. ✅ **手動wrapper実行で成功確認**（20:22）: ラベル解決OK・新着0件・state更新OK・Exit 0
2. ✅ **Task Scheduler 再登録**（20:29）: 
   - 旧: `python.exe` 直接呼び出し / Run As `sshuser`（ejsan領域のpython.exeにアクセス不可）
   - 新: `powershell.exe -ExecutionPolicy Bypass -File run_gmail_thread_watcher.ps1` / Run As `ejsan`
3. ✅ **20:31:15 自動起動成功**: state.json `last_check_at=2026-04-28T20:31:17` 更新、Exit 0、NumberOfMissedRuns=0
4. ✅ **次回 21:01:15 にスケジュール継続**

これにより F8 の問題は完全解消。1106PC は将来の中心的役割拡大に向けて「自律タスクが正しく動く」検証に成功した。

教訓:
- 元の register_task.ps1 は `-User` を指定せず登録者ユーザーで動作するため、誰がスクリプトを実行したかで Run As が変わる「不安定な設計」だった
- 今後の自律タスク登録は **必ず `New-ScheduledTaskPrincipal -UserId` で明示**する

### F9: 廃止と称した方針が実態と乖離

F7 と F8 から学べる構造的問題:
- 「廃止しました」と書いた方針が**実態としては動いていた**（F7: junction）
- 「現役です」と書いたタスクが**実態としては動いていなかった**（F8: gmail watcher）
- どちらも ADR-0004 Reality Check の精神に反する確認不足

これは **ADR/memory に書かれた状態 ≠ 実態** という構造的ドリフトを示す。
今後ADRを書く際は「実機で動作確認した上で」状態を記述する規律が必要（ADR-0004 を強化）。

## Decision（決定）

### D1: 1106PC 役割の正式記述を memory に追加
新規 memory `project_1106pc_active_services.md` を作成し、現役2タスク（Orchestrator + GmailThreadWatcher）を明文化。`project_1106pc_identity.md` も更新。

### D2: 段階的退役（Issue化して順次実施）

「即座に全削除」ではなく、**Issue化→Archive移動→1〜2週間の様子見→本削除**の3段階運用（`project_archive_convention.md` 準拠）。

退役対象を**5グループ**に分類し、それぞれ Issue を1本起票：

- **Issue #22**: ルート直下の廃止アーティファクト退役（HANDOVER / STATUS / session_log / 旧 .ps1 / docs/PC_RESPONSIBILITY 等）
- **Issue #23**: 廃止GASパイプライン痕跡退役（00_Inbox / Scripts/GAS_Chatwork / 04_SE 配下）
- **Issue #24**: memory廃止済記述の退役（handoff_to_*.md / project_dual_pc_sync / project_sync_blockers / reference_status_md）
- **Issue #25**: 1106PC bridge.ps1 等の死蔵スクリプト整理（Phase B0残骸 / process_inbox.py / collect_mf_journals.py の用途確認・退役判断）
- **Issue #26**: 廃止済パス参照の腐敗修正（`ClaudeSync` 等を grep して各文書を更新 or 退役）

### D3: `.claude-memory/` ルート直下コピーの取り扱い決定

選択肢：
- (a) 削除：active memoryのみを正本とし、PC間同期は諦める（ADR-0001準拠）
- (b) 維持：3PC共有の参照用として残す（ただし更新時の二重管理コスト）

→ **(a) 削除** とする。memoryはPC毎独立、ADR-0001の精神を守る。一時的な参照は GitHub の ADR で代替可能。

### D4: 退役運用の標準化

`project_archive_convention.md` を本ADRのリンクで補強し、以下を恒久ルールとする：

- 退役は `_Archive_YYYY-MM-DD/` へ移動（即削除しない）
- 移動から **30日後** に本削除（カレンダー or Issueで自動リマインド検討）
- 各退役は Issue でログ化（commit historyだけでは追跡しにくいため）

### D6: `.claude-memory/` を削除禁止リソースとして明文化（2026-04-28夜 追加）

`.claude-memory/` は3PC全部のmemoryの**唯一の実体**であるため、以下を恒久ルール化する：

- 物理削除（`rm -rf` / GUI削除）禁止
- フォルダ移動禁止（Junctionが切れる）
- 「整理」目的でも触らない
- DESIGN_PRINCIPLES.md 原則2に明記

万一削除した場合の復旧手順（Drive ゴミ箱30日保持の恩恵）も DESIGN_PRINCIPLES.md に明記する。

### D7: ADR-0008 D3（`.claude-memory/`削除）を撤回（2026-04-28夜 追加）

D3 の判断は誤り：
- F7 で発覚した通り、`.claude-memory/` は3PC同期の実装そのもの
- 「(a) 削除：active memoryのみを正本」という判断はそもそも実態誤認
- D3 を**撤回**し、`.claude-memory/` は維持する

代わりに：
- ⚠️ **重複表示の問題は許容**（実体は1つなので問題なし、`.claude-memory/` は Junction Target として必要）
- 削除禁止ルール（D6）で運用

### D5: Out of Scope（今回はやらない）

- 02_CFO/ の中身精査（まだCFO業務で使われている可能性あり、別途棚卸し）
- 06_SE/aieiji_secretary_v2/ の HANDOFF_*.md 等の旧引継書（v3作業の記録として一時残す）
- ADRディレクトリ自体の整理（ADR-0001〜0008 全て生きており不要）
